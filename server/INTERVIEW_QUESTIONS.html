Restaurant Reservation System

Interview Questions & Prepared Answers — Full-Stack Developer Position

Categories: Architecture • Booking Logic • Security • Testing • Deployment • General

Section 1: Architecture & Design Decisions

Q1: Walk me through the architecture of this application.

A: The application follows a standard three-tier architecture:

Frontend (React + Vite): Single-page application with role-based routing. AuthContext manages JWT tokens and provides an Axios client with automatic token injection and 401 interception. ToastProvider handles notifications. ProtectedRoute guards sensitive pages.

Backend (Express + MongoDB): RESTful API with a service-layer pattern. Controllers handle HTTP concerns (request parsing, response formatting). Services encapsulate business logic (booking engine with transaction safety). Middleware handles cross-cutting concerns (auth, validation, error handling, security headers, rate limiting).

Database (MongoDB): Three collections — Users, Tables, Reservations — with compound indexes and schema validation.

The frontend communicates with the backend via REST API over HTTP. In development, Vite proxies /api requests to the Express server. In production, both are deployed separately on Render.

Q2: Why did you choose MongoDB over a relational database?

A: I chose MongoDB for several reasons aligned with the assignment requirements:

  • Schema flexibility: Updating fields is straightforward — the Reservation schema was modified multiple times during development without migrations
  • Document model: Reservation data is naturally hierarchical (user + table + date/timeslot), making embedded documents a good fit
  • ACID transactions: MongoDB 4.0+ supports multi-document transactions, which I used for double-booking prevention
  • Partial unique indexes: The partialFilterExpression on the reservation compound index allows unique constraints on confirmed reservations while allowing cancelled ones to coexist — harder to achieve in SQL without filtered indexes
  • Rapid prototyping: For a 48-hour timeline, MongoDB's schema-less nature allowed faster iteration

However, a relational database (PostgreSQL) would offer stronger referential integrity guarantees and more sophisticated query capabilities for reporting.

Q3: Explain the project structure and how you organized the code.

A: The project follows a separation-of-concerns pattern:

Backend structure:

  • routes/ — Define endpoints, attach validation middleware, wire to controllers
  • controllers/ — Handle HTTP request/response, delegate to services
  • services/ — Business logic (booking engine, availability, cancellation)
  • models/ — Mongoose schemas with validation, indexes, hooks
  • middleware/ — Authentication, error handling, security (Helmet, rate limiting)
  • __tests__/ — Jest test suites organized by module

Frontend structure:

  • context/ — React contexts for shared state (Auth, Toast)
  • components/ — Reusable UI components (Layout, BookingForm, Toast, ProtectedRoute)
  • pages/ — Page-level components (Login, Register, CustomerDashboard, AdminDashboard)

This separation makes the codebase maintainable, testable, and easy to navigate.

Section 2: Booking Logic & Conflict Handling

Q4: How does the double-booking prevention work?

A: I implemented a three-layer defense system:

Layer 1 — Application Check (bookingService.js): Before creating a reservation, the service queries for existing confirmed reservations with the same table, date, and timeSlot. If found, it throws a conflict error immediately.

Layer 2 — Database Index: A unique compound index on {table, date, timeSlot, status} with partialFilterExpression: {status: 'confirmed'}. This means the database itself will reject any attempt to insert a duplicate confirmed reservation, regardless of the application logic.

Layer 3 — Transactions: All booking operations run inside MongoDB ACID transactions with session isolation. If two customers attempt concurrent bookings, one transaction commits while the other is automatically rolled back. All three layers work together — if the application check misses a race condition, the database index catches it, and the transaction ensures clean rollback.

Error handling converts database-level errors (E11000 duplicate key, WriteConflict, TransientTransactionError) into user-friendly 409 Conflict responses.

Q5: How does the availability search work?

A: The findAvailableTables() function works in three steps:

  1. Find booked tables: Query reservations for the requested date range and time slot with status 'confirmed'. Extract the table IDs.
  2. Find available tables: Query tables where _id is NOT in the booked set AND capacity >= numberOfGuests AND status === 'available'.
  3. Sort by capacity: Return smallest tables first — this is a UX optimization that seats smaller groups at appropriately-sized tables, leaving larger tables available for bigger parties.

Cancelled reservations are excluded from the "booked" set since we only query for status 'confirmed', so cancelled slots always appear as available.

Q6: What happens when a reservation is cancelled?

A: Cancellation uses a soft-delete pattern. The reservation's status is changed from 'confirmed' to 'cancelled' — the document is never deleted. This preserves the audit trail and historical data.

Since the unique compound index only applies to documents with status: 'confirmed', cancelled reservations don't block that time slot. The slot becomes available for new bookings immediately.

Authorization checks ensure customers can only cancel their own reservations, while admins can cancel any reservation.

Q7: How did you handle edge cases in the booking logic?

A: I tested several edge cases:

  • Single guest: Capacity 1 table, 1 guest — should succeed
  • Maximum guests: Capacity 20 table, 20 guests — boundary test
  • Table not found: Invalid table ID — returns clear error
  • Overlapping times: Same table, different times — both succeed
  • Different dates: Same table, same time, different dates — both succeed
  • Table in maintenance: Status 'maintenance' tables excluded from availability
  • Duplicate table number: Creating two tables with same number — 409 Conflict

These are covered by the test suite with descriptive test names.

Section 3: Security

Q8: What security measures did you implement?

A: Multiple layers of security:

  • Authentication: JWT tokens signed with a secret, Bearer format required, 7-day expiry
  • Password security: bcrypt hashing with 10 salt rounds, minimum 6 characters
  • Authorization: Role-based middleware (protect + restrictTo(['admin'])) on every protected route
  • Input validation: express-validator on all endpoints + Mongoose schema validation
  • Helmet: Security headers for XSS, clickjacking, MIME-sniffing, HSTS
  • Rate limiting: 100 requests/15min general API, 20/15min on login/register
  • Body size limit: 10kb maximum request payload
  • Environment variables: All secrets in .env, none hardcoded
  • Error handling: No sensitive information leaked in error responses
  • Token management: Automatic 401 handling on frontend, no password in API responses
Q9: How is role-based access control implemented?

A: RBAC is implemented at both the backend and frontend:

Backend: The protect middleware validates the JWT and attaches the user to the request. The restrictTo(['admin']) middleware checks the user's role against the allowed roles. This creates a composable middleware chain — router.post('/', protect, restrictTo(['admin']), createHandler).

Frontend: The AuthContext provides isAuthenticated and isAdmin computed properties. The ProtectedRoute component wraps pages and optionally checks requiredRole. The routing is split: / for customers, /admin for admins, with automatic role-based redirect after login.

API responses: Unauthorized users receive 401 (no token) or 403 (wrong role) with clear error messages.

Section 4: Testing

Q10: What testing strategy did you follow?

A: I wrote 176 tests across 4 test files covering three levels:

  • Unit tests: 62 tests for the booking service, testing individual functions with various inputs and edge cases
  • Integration tests: 114 tests for API endpoints (auth, reservations, tables) using Supertest to make real HTTP requests against the Express app
  • Concurrent tests: 10+ tests using Promise.allSettled to simulate simultaneous booking attempts and verify transaction safety

Tests follow the AAA pattern (Arrange, Act, Assert) with descriptive test names. The database is cleared before each test for isolation. Test utilities provide factory functions for creating test data.

Coverage targets: 80%+ global, 90%+ on the booking service (the core business logic).

Q11: One test fails — how do you explain this?

A: One concurrent booking test fails on this development environment because it requires a 3-node MongoDB replica set to properly detect write conflicts during transaction commits. On a single-node dev replica set, both concurrent transactions can commit simultaneously.

This is a well-documented infrastructure limitation, not a code defect. In production with a proper 3-node replica set, the MongoDB transaction engine would detect the conflict during the commit voting phase and abort one of the transactions. The code logic — unique compound index, ACID transaction, comprehensive error handling — is production-grade and would pass all tests in a clustered environment.

The fix would be to either deploy to a production replica set or configure the local replica set with rs.initiate() and proper read/write concerns.

Section 5: Deployment

Q12: How would you deploy this application?

A: The application is designed for deployment on Render.com with MongoDB Atlas:

  1. Database: Create a free MongoDB Atlas cluster, configure network access (0.0.0.0/0 for Render), create a database user, and get the connection string.
  2. Backend: Render auto-detects the render.yaml Blueprint. It builds the Node.js app using the start script and injects environment variables (MONGO_URI, JWT_SECRET).
  3. Frontend: Render serves the static build from client/dist/. The VITE_API_URL is set to the backend's Render URL.
  4. CI/CD: Push to GitHub main branch triggers automatic redeployment on Render.

Both services are deployed on Render's free tier with automatic HTTPS and custom domain support. The Blueprint configuration and deployment guide are included in the repository.

Section 6: General & Experience

Q13: What was the most challenging part of this project?

A: The most challenging part was ensuring concurrent double-booking prevention across all scenarios. MongoDB transactions on a single-node replica set don't behave identically to a production multi-node setup. I had to:

  1. Research how MongoDB unique indexes interact with transactions
  2. Test various isolation levels (snapshot vs local read concern)
  3. Implement multiple fallback error handlers (E11000, WriteConflict, TransientTransactionError)
  4. Verify the three-layer defense works correctly for sequential, parallel, and edge cases

The frontend-backend integration was also challenging — the Mongoose auto-populate middleware silently transformed ObjectId fields into populated objects, which caused subtle bugs where .toString() returned [object Object] instead of the hex ID. This affected both the availability search (CastError) and the cancellation authorization check (always 403).

A: With more time, I would add:

  3. Real-time updates: WebSocket support so the UI updates when availability changes

  5. Pagination: Paginate reservation lists for better performance with large datasets
  6. Audit logging: Log admin actions for security and compliance
  7. Multi-location support: Allow a single deployment to manage multiple restaurant locations
  8. OAuth: Support Google/GitHub login for easier authentication
