2. Customer Functionality

  • Create Reservation — Select date, time slot (30-min intervals, 11:00-21:00), guest count (1-20), and table from available options
  • View Reservations — See all personal reservations with table number, capacity, date, time, and status
  • Cancel Reservation — Cancel confirmed reservations with confirmation dialog
  • Real-time Availability — Live table search triggered on any input change (date/time/guests)
  • Capacity Validation — Visual feedback when guest count exceeds table capacity
  • Status Indicators — Color-coded badges (green for confirmed, red for cancelled)

3. Administrative Functionality

  • View All Reservations — Complete list with customer details (name, email), table info, and status
  • Filter by Date — Date picker to filter reservations for a specific day
  • Manage Reservations — Update status (confirmed/cancelled) on any reservation
  • Table Management — Create new tables with number and capacity, delete existing tables
  • Admin Navigation — Dedicated "Admin Panel" tab in navigation bar, role-aware route guards
  • Separate Dashboard — Admin interface clearly distinguished from customer view with 2-column layout

4. Booking Engine & Conflict Prevention

4.1 Double-Booking Prevention (3-Layer Defense)

  1. Application-Level Check — Query for existing confirmed reservations with matching table/date/timeSlot before inserting
  2. Database Unique Index — Compound index on {table, date, timeSlot, status} with partialFilterExpression: {status: 'confirmed'} — the ultimate guarantee
  3. ACID Transactions — MongoDB sessions with rollback on any failure, snapshot read concern for isolation

4.2 Capacity Validation

  • Reject if numberOfGuests > table capacity (descriptive error message with both values)
  • Accept if numberOfGuests <= table capacity (including exact match at boundary)
  • Mongoose schema validation with min: 1 and max: 20

4.3 Availability Search

  • Return only tables not booked for the requested date/time slot
  • Filter by capacity (minimum guest requirement)
  • Exclude tables in maintenance mode
  • Sort by capacity ascending (best-fit optimization)
  • Handle cancelled reservations correctly (don't block availability)

4.4 Soft-Delete Cancellation

  • Mark reservations as cancelled — never delete from database
  • Cancelled reservations free up the time slot for new bookings
  • Historical data preserved for audit and reporting

5. Security Features

  • Helmet — Security headers (XSS protection, clickjacking, MIME-sniffing, HSTS)
  • Rate Limiting — 100 requests/15min general API, 20 requests/15min auth endpoints
  • CORS — Configurable origin via environment variable
  • Request Size Limit — 10kb maximum payload
  • bcrypt — Password hashing with 10 salt rounds
  • Input Validation — express-validator on all endpoints + Mongoose schema validation
  • JWT — Tokens signed with secret, Bearer format required, expiry configurable
  • Environment Variables — All secrets via .env, none hardcoded in source

6. API Design

  • RESTful — Resource-based endpoints with proper HTTP methods (GET, POST, PUT, DELETE)
  • HTTP Status Codes — 200, 201, 400, 401, 403, 404, 409, 500 — used appropriately
  • Centralized Error Handling — Error middleware catches all errors, returns consistent JSON format
  • Swagger/OpenAPI — Interactive API documentation at /api-docs
  • Health Check — GET /health endpoint for monitoring
